🗄️ Quorum Replication Explained

🔢

N, W, R Explained

N = total replicas storing the data. W = replicas that must acknowledge a write before it's considered successful. R = replicas consulted on a read — the highest version wins. These three numbers are per-request knobs in Cassandra and DynamoDB.

🔒

Strong Consistency: W + R > N

If W + R > N, the write and read quorums must overlap — at least one node in any read quorum also received the latest write. The read always sees the freshest data. Cassandra calls this QUORUM + QUORUM with N=3, W=2, R=2.

⏳

Eventual Consistency: W + R ≤ N

If W + R ≤ N, write and read quorums may not overlap. A read can hit only nodes that haven't received the latest write yet — returning stale data. The trade-off: lower latency and higher availability during network partitions.

🔧

Read Repair

When a read quorum returns mixed versions, the coordinator returns the highest version to the client and asynchronously sends the latest value back to the stale replicas. This is "read repair" — how Cassandra heals inconsistency without a full sync.
